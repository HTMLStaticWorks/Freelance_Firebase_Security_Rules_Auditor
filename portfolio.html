Vulnerability Analysis

Privilege Escalation

Attackers try to upgrade their account roles (e.g., to 'admin') by including extra fields in an update request.

The Exploit

db.collection('users').doc(myUid).update({
  role: 'admin',
  username: 'attacker'
});
firestore.rules (Defense)
match
/users/{userId} {
allow update: if request.auth.uid == userId
  && !request.resource.data.diff(resource.data).affectedKeys().hasAny(['role']);
}
firestore.rules (Defense)
match
/products/{prodId} {
allow list: if request.query.limit <= 20;
}

Bulletproof Validation

Rules are your first and best defense against malformed data. Use these samples to enforce your schema.

Type & Format Validation

Ensuring incoming data fields match expected types and specific formats (like email regex).

function isValidUser(data) {
  return data.email is string
    && data.email.matches('^.+@.+$')
    && data.age is int
    && data.age >= 18;
}

Field Immutability

Protecting critical fields from being changed after creation (e.g., creation dates, user IDs).

function noFieldChanges(field) {
  return !(field in request.resource.data)
    || request.resource.data[field] == resource.data[field];
}

allow update: if noFieldChanges('createdAt');

Strict Schema Enforcement

Preventing 'ghost' fields by ensuring only allowed keys are present in the incoming resource.

function strictSchema(data, allowedKeys) {
  return data.keys().hasOnly(allowedKeys);
}

allow create: if strictSchema(request.resource.data,
  ['name', 'email', 'bio']);

Relationship Validation

Ensuring references to other documents exist before allowing a document to be created.

allow create: if
  exists(/databases/$(database)/documents/orgs/$(request.resource.data.orgId));
