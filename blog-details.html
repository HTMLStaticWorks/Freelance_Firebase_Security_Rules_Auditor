The Anatomy of a Firestore Privilege Escalation Exploit

A
Alex (FireAudit)
May 12, 2026 8 min read

Role-Based Access Control (RBAC) is the gold standard for managing permissions in complex applications. However, when implemented incorrectly in Firebase, it can leave the door wide open for malicious actors to grant themselves administrative privileges. Here's how it happens.

The Vulnerable Setup

Imagine a standard SaaS application where user profiles are stored in a /users collection. The developer wants users to be able to edit their own profiles, so they write a rule like this:

// Vulnerable Rule
match
/users/{userId} {
allow read, write: if request.auth.uid == userId;
}

At first glance, this looks secure. A user can only write to their own document. But what does the /users/{userId} document actually contain? Often, it contains a role field used by the frontend to determine what UI to show.

The Exploit

Because the rule allows the user to write anything to their document, an attacker doesn't need to hack your servers. They simply open their browser console, instantiate the Firebase SDK, and run this code:

// The Attacker's Script
await firestore.collection('users').doc(myUid).update({
role: 'admin'
});

The Firebase Security Rules evaluate the request: Is the user trying to write to their own document? Yes. The rule passes, the database updates, and the attacker is now an admin.

The Remediation

To fix this, we must shift our thinking from "Who can access this document?" to "Who can modify this specific field?". We need to validate the incoming data using request.resource.data.

Here is the secure implementation:

function
isUnmodified(field) {
return !(field in request.resource.data) ||
       (field in resource.data && resource.data[field] == request.resource.data[field]);
}
match
/users/{userId} {
allow read: if request.auth.uid == userId;
// Ensure the user cannot change their own role
allow write: if request.auth.uid == userId && isUnmodified('role');
}

By using the isUnmodified custom function, we ensure that while the user can update their name or avatar, any attempt to include a modified role field in the payload will result in a PERMISSION_DENIED error.
