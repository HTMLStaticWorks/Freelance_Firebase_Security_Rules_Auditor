What We Fix

Real-world examples of critical vulnerabilities I regularly find and remediate in client databases.

CRITICAL SEVERITY

The "Auth Only" Bypass

Many developers use request.auth != null as a blanket rule. This means ANY logged-in user can read or overwrite ANY other user's data. I rewrite these into strict owner-only logic.

VULNERABLE
// Problem: Any authenticated user has full access
match
/users/{userId} {
allow read, write: if request.auth != null;
}
// Solution: Strictly enforce owner access
match
/users/{userId} {
allow read, write: if request.auth.uid == userId;
}
VULNERABLE
// Problem: Missing schema validation on write
match
/products/{productId} {
allow write: if isAdmin();
}
// Solution: Enforce data types and required fields
match
/products/{productId} {
allow write: if isAdmin() &&
  request.resource.data.price is number &&
  request.resource.data.price >= 0;
}
HIGH SEVERITY

Schema Injection

Without rule-level validation, malicious actors can insert incorrect data types (e.g., strings instead of numbers) that crash your frontend or corrupt the database. I enforce strict schema rules.
